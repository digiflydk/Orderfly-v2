# Brand Website — Schema → UI Mapping

This document describes how the Brand Website configuration schemas map to the Superadmin CMS UI for EPIC 522.

For each field, we document:

- The **schema path**
- The **UI section/tab**
- The **UI label**
- The **input type**
- The **save action** used by the CMS

The mappings below are based on:

- `brandWebsiteConfigBaseSchema`
- `brandWebsiteDesignSystemSchema`
- `brandWebsiteSeoSchema`
- `brandWebsiteSocialSchema`
- `brandWebsiteTrackingSchema`
- `brandWebsiteLegalSchema`

> Note: Field names must always be kept in sync with the actual TypeScript schemas.  
> If a schema is extended, this document must be updated in the same task.

---

## 1. Core Config (brandWebsiteConfigBaseSchema)

These fields define the high-level website configuration that applies to the whole brand website.

**Schema:** `brandWebsiteConfigBaseSchema`  
**Save action:** `saveBrandWebsiteConfig`

| Schema path                | UI section / tab | UI label           | Input type        | Save action                |
|----------------------------|------------------|--------------------|-------------------|----------------------------|
| `config.active`            | Config           | Website active      | Toggle            | `saveBrandWebsiteConfig`   |
| `config.template`          | Config           | Website template    | Select            | `saveBrandWebsiteConfig`   |
| `config.domains[0]`        | Config           | Primary domain      | Text input        | `saveBrandWebsiteConfig`   |
| `config.domains[1..]`      | Config           | Additional domains  | Repeating inputs  | `saveBrandWebsiteConfig`   |
| `config.defaultLocationId` | Config           | Default location    | Location select   | `saveBrandWebsiteConfig`   |

Notes:

- `domains[0]` is treated as the primary domain for public resolution.
- `domains[1..]` are additional domains that also resolve to the same brand website.
- `defaultLocationId` is used by public endpoints to pick a default store/location when needed.

## 2. Design System (brandWebsiteDesignSystemSchema)

The design system stores brand-specific design tokens used by the public Brand Website template.

**Schema:** `brandWebsiteDesignSystemSchema`  
**UI section:** Design System tab  
**Save action:** `saveBrandWebsiteDesignSystem`

| Schema path                               | UI label            | Input type                | Save action                     |

| `designSystem.typography.headingFont`     | Heading font family | Text input or select      | `saveBrandWebsiteDesignSystem` |
| `designSystem.typography.bodyFont`        | Body font family    | Text input or select      | `saveBrandWebsiteDesignSystem` |
| `designSystem.typography.h1Size`          | H1 font size        | Text input (CSS size)     | `saveBrandWebsiteDesignSystem` |
| `designSystem.typography.h2Size`          | H2 font size        | Text input (CSS size)     | `saveBrandWebsiteDesignSystem` |

## 6. Legal (brandWebsiteLegalSchema)

Legal content and platform-default vs custom legal texts.

**Schema:** `brandWebsiteLegalSchema`  
**UI section:** Legal tab  
**Save action:** `saveBrandWebsiteLegal`

| Schema path                 | UI label                           | Input type | Save action              |
|-----------------------------|------------------------------------|-----------|--------------------------|
| `legal.usePlatformDefaults` | Use platform default legal texts   | Toggle    | `saveBrandWebsiteLegal`  |
| `legal.customCookiePolicy`  | Custom cookie policy               | Text area | `saveBrandWebsiteLegal`  |
| `legal.customPrivacyPolicy` | Custom privacy policy              | Text area | `saveBrandWebsiteLegal`  |
| `legal.customTerms`         | Custom terms and conditions        | Text area | `saveBrandWebsiteLegal`  |

> When `usePlatformDefaults` is enabled, custom legal fields may be left empty.  
> When it is disabled, the project may require custom texts according to brand/legal requirements.

---

## 7. Maintenance

Whenever any of the Brand Website schemas are changed:

- `config-schemas.ts` must be updated.
- The CMS UI must be adjusted to expose the new fields.
- This **Schema → UI Mapping** document must be updated in the **same** task so Codex and Studio can rely on it as the single source of truth.
